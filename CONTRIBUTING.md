# Collaboration Guide

## 협업 원칙

WashLog는 해커톤용 빠른 검증 프로젝트입니다. 기능을 크게 나누기보다 사용자가 모바일에서 `등록 → 확인 → 세탁 완료 기록`을 끝낼 수 있는 흐름을 먼저 완성합니다.

## 브랜치와 PR

| 구분 | 규칙 |
|---|---|
| 기본 브랜치 | `main` |
| 작업 브랜치 | `feature/<short-name>`, `fix/<short-name>`, `docs/<short-name>` |
| PR 크기 | 화면, API, 문서처럼 리뷰 가능한 단위로 분리 |
| 머지 기준 | `npm run lint` 통과, 충돌 요구사항은 PR 설명에 명시 |

## PRD 취합 룰

팀원 PRD 원문은 `docs/prd/raw/`에 그대로 보관합니다. 통합 문서에는 중복 설명을 반복하지 않고, 아래 네 가지로만 정리합니다.

| 구분 | 작성 기준 |
|---|---|
| 핵심 판단 | 지금 결정해야 하는 제품 방향 |
| 사용자 요구 | 실제 사용자가 하려는 행동 |
| 근거 | PRD 원문, 인터뷰, 관찰에서 나온 이유 |
| 다음 액션 | 구현, 검증, 보류 중 하나로 분류 |

서로 다른 PRD가 충돌하면 한쪽을 숨기지 않고 `결정 필요`로 남깁니다.

## 코드 리뷰 기준

모바일 화면에서 버튼을 한 손으로 누를 수 있는지, 기록이 3초 안에 끝나는지, 빈 상태와 오류 상태가 있는지를 먼저 봅니다. 구현 세부보다 사용 흐름이 끊기는 문제가 우선입니다.

## 보안

Public repo 기준으로 작업합니다. `.env`, Supabase 키, 개인 세탁 기록 샘플, 배포 토큰은 커밋하지 않습니다. 필요한 환경 변수는 `.env.example`에 이름만 남깁니다.
