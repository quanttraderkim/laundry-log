# WashLog 고도화 PRD

작성일: 2026-05-19  
상태: Draft  
기준 문서: `docs/PRD.md`, `docs/DATABASE_DECISION.md`, `docs/prd/PHOTO_REGISTRATION_PRD.md`

## 1. 결론

WashLog의 다음 고도화 방향은 단순한 `착용/세탁 기록장`이 아니라, 옷별 상태를 보고 다음 관리 행동을 결정해주는 `의류 관리 의사결정 앱`입니다. 기존 MVP는 마지막 착용일, 마지막 세탁일, 착용 횟수, 소재별 권장 이유를 이미 보여주므로, 다음 단계에서는 기록을 이벤트 히스토리로 남기고 `다음 행동`, `권장 날짜`, `권장 근거`, `사용자 수정`까지 닫힌 루프로 만들어야 합니다.

| 항목 | 결정 |
|---|---|
| 1차 고도화 목표 | 옷별 `다음 관리 행동`을 추천하고, 사용자가 실행/수정/보류할 수 있게 함 |
| 차별화 축 | 일반 옷장 앱의 코디/통계가 아니라 세탁, 통풍, 스타일러, 드라이클리닝 판단에 집중 |
| 우선 사용자 | 운동복과 출근복, 니트/아우터를 반복 착용하는 1인 사용자 |
| 핵심 화면 | 첫 화면의 `오늘 관리할 옷`, 옷 상세의 `관리 히스토리`, 빠른 기록의 `날짜 보정` |
| 저장 전략 | 데모는 LocalStorage 유지, 고도화 시 Supabase Postgres 이벤트 로그로 전환 |
| 별도 PRD | 사진 기반 옷 등록은 `PHOTO_REGISTRATION_PRD.md`의 P1 흐름으로 분리 유지 |

## 2. 외부 조사 요약

의류별 세탁 주기는 일괄 규칙으로 정하기 어렵고, 소재, 활동량, 피부 접촉, 오염 상태, 관리 라벨에 따라 달라집니다. 따라서 WashLog는 “정답”을 말하기보다 “현재 정보 기준의 보수적 제안”을 보여주고 사용자가 수정할 수 있어야 합니다.

| 근거 | 제품 반영 |
|---|---|
| American Cleaning Institute는 활동, 옷감, 오염도, 날씨에 따라 세탁 필요 여부가 달라진다고 설명하고, 속옷/양말/운동복/티셔츠는 잦은 세탁, 청바지는 여러 번 착용 후 세탁을 권장합니다. | 소재와 카테고리만으로 고정하지 않고 `땀`, `냄새`, `얼룩`, `피부 접촉` 플래그를 우선 반영 |
| Applied and Environmental Microbiology 연구는 운동 후 폴리에스터 의류가 면보다 냄새가 더 강하고 불쾌하게 평가됐다고 보고합니다. | 폴리에스터 운동복은 `1회 착용 후 세탁 권장`을 기본값으로 둠 |
| Woolmark는 울이 냄새, 오염, 구김에 강해 세탁 빈도를 줄일 수 있고, 세탁 전 관리 라벨 확인을 강조합니다. | 울/니트는 `세탁 권장`보다 `통풍/부분 관리`를 먼저 제안 |
| FTC Care Labeling Rule은 의류 관리 라벨이 일반 사용에 필요한 관리 절차와 경고를 제공해야 한다고 봅니다. | 앱 추천은 라벨을 대체하지 않고, 사용자가 `관리 라벨 제약`을 입력하면 앱 규칙보다 우선 적용 |
| WearTracker, MODA, Wearli 같은 옷장 앱은 착용 기록, 사진 등록, 자동 태깅, 날씨/코디 추천, Cost per Wear에 강점이 있습니다. | WashLog는 코디 추천보다 `세탁 가능 상태`, `관리 필요 이유`, `다음 관리 행동`에 집중해 포지션을 분리 |

참고: [ACI 세탁 빈도 가이드](https://www.cleaninginstitute.org/cleaning-tips/clothes/laundry-basics/do-i-need-wash), [Applied and Environmental Microbiology 연구](https://journals.asm.org/doi/10.1128/AEM.01422-14), [Woolmark 울 관리](https://www.woolmark.com/care/how-to-wash-wool-sweater/), [FTC Care Labeling Rule](https://www.ftc.gov/legal-library/browse/rules/care-labeling-textile-wearing-apparel-certain-piece-goods-text), [WearTracker](https://apps.apple.com/us/app/weartracker-closet-organizer/id1371856481), [MODA](https://www.getmoda.app/), [Wearli](https://www.wearli.app/en)

## 3. 현재 MVP 진단

현재 앱은 모바일 첫 화면에서 등록한 옷, 관리 필요 수, 오늘 기록 수를 보여주고, 카드별로 마지막 착용일과 마지막 세탁일을 절대 날짜와 상대 날짜로 표시합니다. 또한 `오늘 입음`, `스타일러`, `세탁 완료`, `땀남/냄새/얼룩` 기록과 빠른 문장 입력, 브랜드/종류 필터, 정렬, LocalStorage 저장까지 갖추고 있습니다.

고도화 여지는 세 가지입니다. 첫째, 기록이 현재 상태 필드에 가까워서 과거 히스토리, 수정, 되돌리기, 날짜 보정이 어렵습니다. 둘째, 추천은 카드 상태로 보이지만 “다음에 무엇을 언제 해야 하는지”가 독립된 할 일로 정리되지는 않습니다. 셋째, 사용자가 추천을 수락했는지, 무시했는지, 규칙을 바꿨는지 학습하는 구조가 없습니다.

## 4. 제품 목표

| 목표 | 성공 기준 |
|---|---|
| 관리 판단을 빠르게 함 | 첫 화면에서 오늘 관리할 옷과 이유를 5초 안에 파악 |
| 기록 신뢰도를 높임 | 착용/세탁/통풍/스타일러 기록을 날짜별 히스토리로 확인, 수정, 삭제 가능 |
| 추천을 개인화함 | 사용자가 추천을 보류하거나 기준을 바꾸면 이후 추천에 반영 |
| 등록 장벽을 낮춤 | 사진 등록 PRD와 연결해 이름/색상/카테고리 입력 부담을 줄임 |
| DB 전환 기반을 만듦 | LocalStorage 모델을 이벤트 로그 구조로 맞춰 Supabase 전환 비용을 낮춤 |

## 5. 범위

### P0. 다음 관리 행동 엔진

각 옷은 현재 상태 라벨과 별도로 `next_action`을 가집니다. 가능한 행동은 `세탁`, `통풍`, `스타일러`, `드라이 점검`, `상태 확인`, `바로 착용 가능`입니다. 추천에는 `권장 날짜`, `권장 이유`, `근거 신호`, `신뢰도`를 함께 둡니다.

| 입력 신호 | 예시 | 우선순위 |
|---|---|---|
| 상태 플래그 | 땀, 냄새, 얼룩 | 가장 높음 |
| 관리 라벨 제약 | 손세탁만, 드라이클리닝, 건조기 금지 | 높음 |
| 소재/카테고리 규칙 | 폴리에스터 운동복, 울 니트, 데님 | 중간 |
| 착용 횟수와 경과일 | 세탁 후 3회 착용, 마지막 세탁 14일 전 | 중간 |
| 사용자 선호 | 나는 데님은 5회까지 입음 | 중간 |
| 계절/보관 상태 | 시즌 종료 코트 | 낮음 |

### P0. 이벤트 히스토리와 날짜 보정

모든 행동은 `clothing_events`로 남깁니다. 사용자는 오늘 기록뿐 아니라 어제 입은 옷, 지난주 세탁한 옷도 캘린더 날짜로 보정할 수 있어야 합니다. 카드에는 최신 상태를 보여주고, 상세에서는 타임라인을 보여줍니다.

| 기능 | 요구사항 |
|---|---|
| 이벤트 추가 | 착용, 세탁, 통풍, 스타일러, 드라이, 상태 메모를 날짜와 함께 저장 |
| 이벤트 수정 | 날짜, 행동, 상태 플래그 수정 가능 |
| 이벤트 삭제 | 잘못 누른 기록 삭제 가능 |
| 상세 타임라인 | `2026.05.19 착용`, `2026.05.18 세탁`처럼 절대 날짜 기준 표시 |
| 상태 재계산 | 이벤트 변경 시 착용 횟수, 마지막 세탁일, 다음 행동을 즉시 재계산 |

### P0. 오늘 관리할 옷

첫 화면 상단에 `오늘 관리할 옷` 영역을 둡니다. 단순히 상태가 나쁜 옷을 모으는 것이 아니라 실행 가능한 행동 기준으로 묶습니다.

| 그룹 | 노출 기준 | 대표 액션 |
|---|---|---|
| 세탁 필요 | 땀/냄새/얼룩, 규칙 초과 | 세탁 완료 |
| 통풍 권장 | 울/아우터 착용 후 세탁 전 관리 | 통풍 완료, 스타일러 |
| 상태 확인 | 기준 근접, 오래 보관, 흰옷/실크 | 확인 완료, 메모 |
| 바로 착용 가능 | 깨끗하고 기준 여유 | 오늘 입음 |

### P1. 사용자 기준 편집

추천이 불편하면 사용자가 옷별 기준을 바꿀 수 있어야 합니다. 예를 들어 데님은 3회가 아니라 5회 착용 후 점검으로 바꾸거나, 특정 셔츠는 매 착용 후 세탁으로 바꿀 수 있습니다.

| 설정 | 예시 |
|---|---|
| 착용 횟수 기준 | 1회, 2회, 3회, 5회 |
| 경과일 기준 | 3일, 7일, 14일, 시즌 종료 |
| 기본 관리 방식 | 세탁기, 손세탁, 통풍, 스타일러, 드라이 |
| 피부 접촉 정도 | 직접 닿음, 레이어 위 착용, 아우터 |
| 라벨 제약 | 건조기 금지, 손세탁, 드라이클리닝 |

### P1. 세탁 바구니와 일괄 처리

관리 필요 옷을 하나씩 누르는 대신 `세탁 바구니`에 담고, 실제 빨래가 끝나면 여러 벌을 한 번에 `세탁 완료` 처리합니다. 해커톤 이후 실제 사용성을 크게 높이는 기능입니다.

### P1. 알림 준비

해커톤에서는 푸시 알림까지 구현하지 않아도 됩니다. 대신 알림 모델을 준비합니다. 사용자가 앱을 열었을 때 `오늘 할 일`로 보이고, 이후 Web Push 또는 이메일로 확장합니다.

### P2. 옷장 인사이트

착용 통계, Cost per Wear, 자주 안 입는 옷, 계절별 보관, 코디 추천은 P2로 둡니다. 경쟁 옷장 앱들이 이미 강한 영역이므로 WashLog의 핵심 관리 루프가 검증된 뒤 붙입니다.

## 6. 추천 규칙 초안

추천은 안전한 의류 관리 힌트이며, 사용자의 세탁 라벨과 실제 오염 상태가 더 우선합니다.

| 프로필 | 기본 다음 행동 | 기본 기준 | 예외 |
|---|---|---|---|
| `active_synthetic` | 세탁 | 1회 착용 또는 땀/냄새 기록 | 냄새 없음으로 사용자가 보류하면 다음엔 상태 확인으로 낮춤 |
| `daily_skin` | 세탁 | 매 착용 후 | 속옷, 양말, 이너웨어 |
| `cotton_top` | 세탁 또는 상태 확인 | 1~2회 착용 | 흰옷/얼룩은 즉시 관리 |
| `standard_outer` | 상태 확인 | 2~3회 착용 | 땀/냄새/비 맞음은 세탁 또는 통풍 |
| `denim` | 상태 확인 | 3회 착용 또는 14일 경과 | 사용자가 기준을 5회로 변경 가능 |
| `wool_delicate` | 통풍 | 착용 후 통풍 우선 | 얼룩/강한 냄새는 라벨 확인 후 세탁/드라이 |
| `seasonal_outer` | 통풍 또는 드라이 점검 | 착용 후 통풍, 시즌 종료 시 점검 | 비/음식 냄새 기록 시 우선 관리 |

## 7. 핵심 사용자 흐름

| 상황 | 흐름 | 수용 기준 |
|---|---|---|
| 어제 입은 셔츠를 오늘 기록 | 카드에서 `오늘 입음` 옆 날짜 선택, 어제 날짜 지정 | 마지막 착용일이 절대 날짜로 반영되고 히스토리에 남음 |
| 러닝 후 운동복 기록 | `나이키 러닝 반팔 입고 땀 났어` 입력 | 착용 이벤트와 땀 플래그가 저장되고 다음 행동이 세탁으로 변경 |
| 울 니트 착용 | `오늘 입음` 선택 | 세탁이 아니라 통풍 또는 스타일러를 먼저 제안 |
| 여러 벌 빨래 완료 | 세탁 바구니에서 4벌 선택 후 `세탁 완료` | 선택한 옷의 마지막 세탁일과 착용 횟수가 일괄 초기화 |
| 추천이 맞지 않음 | 카드에서 `기준 변경` 선택 | 해당 옷의 추천 기준이 저장되고 다음 계산부터 반영 |

## 8. 데이터 모델

LocalStorage에서도 아래 구조와 비슷하게 저장하면 Supabase 이전이 쉬워집니다. 핵심은 `현재 필드 덮어쓰기`가 아니라 `이벤트 추가 후 파생 상태 계산`입니다.

| 테이블/컬렉션 | 주요 필드 |
|---|---|
| `clothing_items` | `id`, `user_id`, `name`, `brand`, `category`, `material`, `color`, `feature`, `size`, `care_profile`, `skin_contact_level`, `care_label_constraints`, `created_at`, `archived_at` |
| `clothing_events` | `id`, `clothing_item_id`, `event_type`, `event_date`, `care_method`, `condition_flags`, `source`, `raw_input`, `note`, `created_at`, `updated_at` |
| `care_rules` | `id`, `profile`, `default_action`, `wear_threshold`, `days_threshold`, `priority`, `reason_template` |
| `item_rule_overrides` | `id`, `clothing_item_id`, `wear_threshold`, `days_threshold`, `preferred_action`, `muted_until`, `created_at` |
| `recommendation_feedback` | `id`, `clothing_item_id`, `recommendation_id`, `feedback_type`, `created_at` |

추천 결과는 처음에는 매번 계산해도 됩니다. 사용자가 추천을 수락/보류/수정한 이력만 저장하고, 운영 단계에서 필요해지면 `recommendation_snapshots`를 추가합니다.

## 9. 화면 요구사항

첫 화면은 지금처럼 모바일 목록을 유지하되, 카드 위에 `오늘 관리할 옷`을 둡니다. 카드에는 현재처럼 절대 날짜와 상대 날짜를 함께 보여주고, 상세 진입 시 캘린더형 히스토리를 제공합니다.

| 화면 | 요구사항 |
|---|---|
| 홈 | 오늘 관리할 옷, 전체 상태 요약, 빠른 기록, 필터/정렬 |
| 옷 카드 | 다음 행동, 권장 날짜, 마지막 착용일, 마지막 세탁일, 이유 |
| 옷 상세 | 히스토리, 기준 설정, 라벨 제약, 메모 |
| 세탁 바구니 | 여러 벌 선택, 일괄 세탁 완료, 일괄 통풍 완료 |
| 기준 편집 | 추천 기준 변경, 기본값 복원, 변경 이유 짧은 안내 |

| 지표 | 목표 |
|---|---|
| 기록 완료 시간 | 카드 액션 기준 3초 이내, 문장 입력 기준 8초 이내 |
| 추천 이해도 | 사용자 5명 테스트에서 `왜 이 행동이 뜨는지` 4명 이상 설명 가능 |
| 추천 수락률 | 관리 필요 추천 중 50% 이상이 실행 또는 보류 액션으로 처리 |
| 수정률 | 추천 기준 수동 변경이 30%를 넘으면 기본 규칙 재검토 |
| 재방문 | 데모 사용자 D7 재방문 30% 이상 |

## 11. 구현 순서

| 순서 | 범위 | 산출물 |
|---|---|---|
| 1 | 이벤트 모델을 LocalStorage에 도입 | `clothing_events` 기반 상태 계산 |
| 2 | 다음 관리 행동 엔진 분리 | 테스트 가능한 `careRules`/`recommendation` 모듈 |
| 3 | 홈에 오늘 관리할 옷 추가 | 행동별 그룹과 일괄 액션 |
| 4 | 옷 상세와 히스토리 추가 | 캘린더 날짜 기반 타임라인, 수정/삭제 |
| 5 | 사용자 기준 편집 | 옷별 override 저장 |
| 6 | Supabase 전환 | Auth, RLS, Postgres schema, 마이그레이션 |

## 13. 다음 액션

이 PRD 기준으로 바로 구현한다면 `이벤트 히스토리 + 다음 관리 행동`을 첫 번째 개발 이슈로 자르는 것이 좋습니다. 이유는 현재 UI와 가장 잘 이어지고, Supabase로 넘어갈 때도 버릴 코드가 적기 때문입니다. 사진 등록, Cost per Wear, 코디 추천은 매력적이지만 WashLog의 핵심 차별화가 검증된 다음 붙이는 편이 낫습니다.
